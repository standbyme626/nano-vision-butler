## 2. 当前现状（与仓库对齐）

1. 架构边界已明确：RK3566 负责边缘感知，后端负责 state/policy 真相层，MCP/Skill/Telegram 负责编排与解释（见 `README.md`）。
2. 主链已存在：`observation -> event -> state -> freshness` 的路由与服务均已落地（见 `src/services/*` 与 `README.md`）。
3. Edge 协议已冻结并校验：`edge.event.v1` schema 已落地，入站校验已在 `perception_service` 执行（见 `schemas/edge_event_envelope.schema.json`、`src/services/perception_service.py`）。
4. OCR 后端触发已具备：`analysis_requests` 已支持 `ocr_quick_read`/`ocr_extract_fields`，并写审计日志（见 `src/services/perception_service.py`）。
5. staleness/fallback 已具备：`evaluate_staleness` 已返回 `reason_code` 与 `fallback_required`（见 `src/services/policy_service.py`）。

## 3. 从修复1保留的高价值原则

1. 不再把复杂能力继续堆到 RK3566；优先“稳定供数”。
2. 以事件质量优先于 FPS：先做去抖、去重、节流和证据链。
3. 强化状态层：`object_state / zone_state / world_state` 要成为回答主依据。
4. 强化 freshness 输出：回答必须带“结论 + 依据 + 证据时间 + 新鲜度/不确定性”。
5. 固定问题到工具链映射，减少模型自由发挥。

## 4. 字段与结构修订（对源文档纠偏）

1. 事件对象字段以当前 schema 为准：`object_class`/`confidence`，不是 `class_name`/`score`。
2. MCP tools 当前集中在 `src/mcp_server/tools/registry.py`，不是多个 `tools/*.py` 功能文件。
3. `analysis_requests` 当前已落地 OCR 两类；扩展到更多类型应作为增量，不应重做现有链路。
4. 默认 RKNN 模型以 `scripts/start_edge.sh` 为准：`yolov8n_rockchip_opt_i8_rk3566.rknn`。

## 5. 第二阶段行动包（收敛优先级）

### A. Edge 稳定事实源（P0）

1. 稳定输出 envelope：字段、时间戳、`track_id`、`zone_id`、`snapshot_uri` 一致。
2. 强化事件压缩策略配置化（阈值/去重/节流），避免每帧噪声事件。
3. 统一分段性能日志（capture/infer/postprocess/emit）用于定位瓶颈。

验收：
- 长时间运行稳定；
- event 质量稳定（低重复、可追踪）；
- snapshot/clip/heartbeat 链路稳定。

### B. 后端状态与策略可信化（P0）

1. 强化 `state_service` 的状态推导质量与证据绑定。
2. 强化 `policy_service` 的 freshness 判定口径与 reason code 可解释性。
3. 保证 object/zone/world 查询返回一致结构，便于 MCP/Skill 消费。

验收：
- “最后一次在哪里出现”“现在大概率还在不在”“区域像不像有人/有物”回答稳定；
- 返回始终携带证据时间、新鲜度、reason code。

### C. 轻前端重后端的通用分析闭环（P1）

1. 在保留 OCR 现有能力前提下，评估将 `analysis_requests` 扩展为通用后端分析入口。
2. 将“提示触发 -> 后端执行 -> 写回状态/审计”的流程模板化，避免特例分支膨胀。

验收：
- 分析触发路径统一；
- 审计中可区分普通事件与后端分析事件。

### D. MCP/Skill/Telegram 话术收口（P1）

1. 固定问题到工具链映射：
   - 现在看到了什么：`take_snapshot -> describe_scene`
   - 最近发生了什么：`query_recent_events`
   - 最后一次在哪里：`last_seen_object`
   - 大概率还在不在：`get_object_state + evaluate_staleness`
   - 区域像不像有人/有物：`get_zone_state`
2. 统一回复结构：结论、依据、证据时间、新鲜度/不确定性。

验收：
- 同类问题走同一路径；
- 输出稳定可解释，不依赖模型临场拼接。

## 6. 暂不优先项

1. 不优先新增 MCP 服务拆分进程。
2. 不优先扩展板端复杂 OCR/重分析。
3. 不优先追求单一 FPS 数字，优先链路稳定性和状态可信度。
